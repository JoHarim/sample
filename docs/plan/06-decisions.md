# 결정 기록 — 스마트 알람 (가칭)
> 방향을 정하거나 바꾼 결정을 위에서부터 쌓는다(최신이 위). 한 결정 = 4줄.
> 왜 남기나: 2주 뒤의 나(와 새 AI 세션)가 "왜 이렇게 만들었지?"를 다시 고민하지 않게.

## [2026-07-09] 울림 화면은 "알림 수신/탭으로 진입", 잠금화면 풀스크린은 제외(MVP)
- 왜: 잠긴 화면 위에 꽉 찬 알람을 띄우는 건 안드로이드 풀스크린 인텐트 등 네이티브 영역이라 Expo Go 기본으로는 불가. MVP는 "알림이 뜸 → 앱이 켜져 있으면 바로, 꺼져 있으면 알림 탭 → 울림 화면"으로 충분.
- 버린 대안: 잠금화면 위 풀스크린 알람 — 커스텀 네이티브 빌드(dev client)와 추가 권한이 필요, 첫 앱 스코프 밖.
- 영향: RingScreen은 알림 수신(포그라운드)·응답(탭) 리스너로 진입. 콜드스타트(앱 완전 종료 후 탭)는 오래된 응답 오작동 위험이 있어 미사용. 로드맵 "완성 제품"으로 보존.

## [2026-07-09] 알람 소리는 코드로 생성한 삐- 음(alarm.wav) 사용
- 왜: 외부 음원 파일이 없고 저작권·용량 부담을 피하려 22kHz 모노 삐-삐- 패턴을 스크립트로 생성해 번들. expo-av로 반복 재생.
- 버린 대안: 시스템 기본 알림음만 사용 — 알림이 한 번만 울리고 울림 화면에선 무음이라 알람답지 않음.
- 영향: apps/mobile/assets/alarm.wav(약 52KB) 번들. 소리 커스텀(사용자 음원 선택)은 다음 버전.

## [2026-07-07] 알림 권한은 S0 온보딩 화면 대신 앱 시작 시 요청으로 대체(MVP)
- 왜: S0 온보딩의 나머지 반쪽(도시 설정)이 블럭 3 소관이라, 지금 화면을 만들면 반쪽짜리가 된다. 권한 요청은 시작 시 자동으로도 충분하고, 거부 시 S1 경고 배너가 안내를 이어받는다.
- 버린 대안: S0 온보딩 화면을 지금 구현 — 블럭 3에서 도시 설정과 함께 완성하는 게 한 번에 끝난다.
- 영향: S0 화면은 블럭 3에서 구현. 앱 시작 시 권한 팝업 + 거부 시 배너(설정으로 이동, 돌아오면 자동 재확인).

## [2026-07-07] 알림 예약은 "전체 취소 후 재예약" + 직렬화 큐로
- 왜: 알람 수가 적어 장부를 통째로 갈아끼우는 게 개별 예약 ID 추적보다 단순하고, 직렬화 큐(최신 목록만 반영)가 동시 실행 때 끈 알람이 남는 사고를 막는다.
- 버린 대안: 알람별 예약 ID를 저장해 개별 취소 — 장부 두 개(알람·예약ID)를 맞춰야 해 어긋날 틈이 더 많다.
- 영향: notifications.ts에 큐 로직. 시작 시·변경 시 재예약으로 재부팅 유실도 자연 복구. 안드로이드 정확 알람 권한(SCHEDULE/USE_EXACT_ALARM)을 app.json에 선언.

## [2026-07-07] 폰 로컬 저장은 SQLite 대신 AsyncStorage(키-값 메모장)로
- 왜: 알람은 수십 개 수준의 작은 목록이라 통째 JSON 저장이면 충분하다. 표 데이터베이스(SQLite)는 이 크기엔 과하고 코드가 배로 늘어난다(최소 코드 원칙).
- 버린 대안: expo-sqlite — 05-db.md의 SQL 초안 그대로 쓸 수 있지만, 스키마·쿼리 관리 비용이 이 규모에선 낭비. 데이터가 커지거나 검색이 생기면 그때 이사.
- 영향: src/storage.ts에 loadAlarms/saveAlarms 2함수. 키 "alarms:v1"에 버전을 박아 나중에 모양이 바뀌면 v2로 안전하게 이사 가능. 05-db.md의 SQL 초안은 다음 버전(서버 전환)용으로 유지.

## [2026-07-05] 화면 전환은 네비게이션 라이브러리 없이 상태(state) 기반으로
- 왜: 화면이 목록<->추가 정도로 적고, S3(울림)은 알림에서 열리고 S0(온보딩)은 첫 실행 조건부 렌더라 스택 네비게이션이 불필요. 부품 최소화·번들 안정.
- 버린 대안: react-navigation / expo-router — 표준이지만 네이티브 모듈·설정이 늘고 이 규모엔 과함.
- 영향: App.tsx의 screen 상태로 분기. 안드로이드 상단 여백은 별도 라이브러리 대신 StatusBar.currentHeight로 처리. 화면이 크게 늘면 그때 재검토.

## [2026-07-05] pnpm을 hoisted(평평하게) 설치로 전환 — 루트 .npmrc node-linker=hoisted
- 왜: React Native/Expo 번들러(Metro)가 pnpm 기본(격리·symlink) 배치에서 @babel/runtime 등 부품을 못 찾아 번들이 실패했다.
- 버린 대안: @babel/runtime을 apps/mobile deps로만 추가 — symlink는 걸렸지만 Metro가 못 따라가 여전히 실패. per-package로는 해결 안 됨.
- 영향: 루트 .npmrc 추가, node_modules 전체 재설치(hoisted). web 포함 전 워크스페이스가 평평 배치로 설치됨. 이후 expo export Android 번들 성공(533모듈).

## [2026-07-05] 웹 대신 모바일 앱(Expo)으로 만든다
- 왜: 알람은 폰이 잠겨 있어도 정해진 시각에 울려야 하는데, 웹(브라우저)은 폰이 잠들면 같이 멈춰 신뢰성 있는 알람이 불가능하다.
- 버린 대안: 웹앱(apps/web) — 개발은 빠르지만 "실제로 깨우는" 핵심 가치를 못 낸다.
- 영향: apps/mobile(Expo) 사용, pnpm-workspace에서 mobile 활성화 필요. 배포는 Vercel이 아니라 Expo EAS Build → 앱스토어/구글플레이.

## [2026-07-05] MVP는 폰 로컬에만 저장(Supabase 서버 미사용)
- 왜: 로그인·기기 동기화가 다음 버전이고, 수면·위치 민감정보를 최소화하려면 데이터를 폰 밖으로 안 내보내는 게 가장 안전·간단.
- 버린 대안: 처음부터 Supabase 서버 저장 — 로그인·설정 작업이 늘고 개인 무료 앱엔 과함.
- 영향: 05-db는 로컬 SQLite 구조. 서버 RLS 불필요. 기기 분실 시 데이터 손실(다음 버전 동기화로 해결).

## [2026-07-05] 실제 수면 감지 스마트웨이크는 MVP에서 제외
- 왜: 밤새 폰 센서로 수면 단계를 추정하는 건 배터리·백그라운드 제약·정확도 문제로 첫 앱 스택의 감당선 밖(전문 영역).
- 버린 대안: MVP에 수면 감지 포함 — 위험이 크고 완성 못 할 확률이 높다.
- 영향: 로드맵 "완성 제품" 층으로 보존. 대체 경로(직접 공부/외주)는 나중에 선택.

## [2026-07-05] 앱 정체성 = "똑똑한 시간" + "확실히 깨우기" 둘 다
- 왜: 사용자의 불편(무의식중에 꺼버림)은 확실히 깨우기가, 차별점(상황 반영)은 똑똑한 시간이 해결 — 둘 중 하나만으로는 문제를 다 못 푼다.
- 버린 대안: 하나에만 집중(시간만 / 깨우기만) — 더 좁지만 이 사용자의 문제를 절반만 해결.
- 영향: MVP 3개 = 기본 알람 + 날씨 조정 + 밀어서 끄기로 두 축을 모두 담음.

## [2026-07-05] 스누즈 없음
- 왜: "확실히 깨우기"가 목표. 스누즈는 다시 잠드는 통로.
- 버린 대안: 기본 스누즈 제공 — 익숙하지만 취지와 충돌. (스누즈 제한은 다음 버전 검토)
- 영향: S3 울림은 밀어서 끄면 완전 종료. 상태 흐름 단순.

## [2026-07-05] 날씨 위치는 도시 직접 설정(GPS 미사용)
- 왜: 민감정보(위치) 최소화. 도시명만 있으면 날씨 조회 충분.
- 버린 대안: 현재 위치 자동(GPS) — 편하지만 위치 권한·위치정보 취급이 늘어남 → 다음 버전.
- 영향: 위치 권한 불필요. settings.city에 도시명만 저장, 날씨 API엔 도시명만 전송.

## [2026-07-05] 날씨 API = OpenWeatherMap
- 왜: 무료 티어 있고 연결이 간단해 바이브코딩 첫 연동으로 무난.
- 버린 대안: 기상청 단기예보 API — 한국 정확도는 높지만 신청·규격이 복잡.
- 영향: .env에 OPENWEATHER_API_KEY 하나. 도시명 기반 조회.
