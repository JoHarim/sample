# 결정 기록 — 스마트 알람 (가칭)
> 방향을 정하거나 바꾼 결정을 위에서부터 쌓는다(최신이 위). 한 결정 = 4줄.
> 왜 남기나: 2주 뒤의 나(와 새 AI 세션)가 "왜 이렇게 만들었지?"를 다시 고민하지 않게.

## [2026-07-05] pnpm을 hoisted(평평하게) 설치로 전환 — 루트 .npmrc node-linker=hoisted
- 왜: React Native/Expo 번들러(Metro)가 pnpm 기본(격리·symlink) 배치에서 @babel/runtime 등 부품을 못 찾아 번들이 실패했다.
- 버린 대안: @babel/runtime을 apps/mobile deps로만 추가 — symlink는 걸렸지만 Metro가 못 따라가 여전히 실패. per-package로는 해결 안 됨.
- 영향: 루트 .npmrc 추가, node_modules 전체 재설치(hoisted). web 포함 전 워크스페이스가 평평 배치로 설치됨. 이후 expo export Android 번들 성공(533모듈).

## [2026-07-05] 웹 대신 모바일 앱(Expo)으로 만든다
- 왜: 알람은 폰이 잠겨 있어도 정해진 시각에 울려야 하는데, 웹(브라우저)은 폰이 잠들면 같이 멈춰 신뢰성 있는 알람이 불가능하다.
- 버린 대안: 웹앱(apps/web) — 개발은 빠르지만 "실제로 깨우는" 핵심 가치를 못 낸다.
- 영향: apps/mobile(Expo) 사용, pnpm-workspace에서 mobile 활성화 필요. 배포는 Vercel이 아니라 Expo EAS Build → 앱스토어/구글플레이.

## [2026-07-05] MVP는 폰 로컬에만 저장(Supabase 서버 미사용)
- 왜: 로그인·기기 동기화가 다음 버전이고, 수면·위치 민감정보를 최소화하려면 데이터를 폰 밖으로 안 내보내는 게 가장 안전·간단.
- 버린 대안: 처음부터 Supabase 서버 저장 — 로그인·설정 작업이 늘고 개인 무료 앱엔 과함.
- 영향: 05-db는 로컬 SQLite 구조. 서버 RLS 불필요. 기기 분실 시 데이터 손실(다음 버전 동기화로 해결).

## [2026-07-05] 실제 수면 감지 스마트웨이크는 MVP에서 제외
- 왜: 밤새 폰 센서로 수면 단계를 추정하는 건 배터리·백그라운드 제약·정확도 문제로 첫 앱 스택의 감당선 밖(전문 영역).
- 버린 대안: MVP에 수면 감지 포함 — 위험이 크고 완성 못 할 확률이 높다.
- 영향: 로드맵 "완성 제품" 층으로 보존. 대체 경로(직접 공부/외주)는 나중에 선택.

## [2026-07-05] 앱 정체성 = "똑똑한 시간" + "확실히 깨우기" 둘 다
- 왜: 사용자의 불편(무의식중에 꺼버림)은 확실히 깨우기가, 차별점(상황 반영)은 똑똑한 시간이 해결 — 둘 중 하나만으로는 문제를 다 못 푼다.
- 버린 대안: 하나에만 집중(시간만 / 깨우기만) — 더 좁지만 이 사용자의 문제를 절반만 해결.
- 영향: MVP 3개 = 기본 알람 + 날씨 조정 + 밀어서 끄기로 두 축을 모두 담음.

## [2026-07-05] 스누즈 없음
- 왜: "확실히 깨우기"가 목표. 스누즈는 다시 잠드는 통로.
- 버린 대안: 기본 스누즈 제공 — 익숙하지만 취지와 충돌. (스누즈 제한은 다음 버전 검토)
- 영향: S3 울림은 밀어서 끄면 완전 종료. 상태 흐름 단순.

## [2026-07-05] 날씨 위치는 도시 직접 설정(GPS 미사용)
- 왜: 민감정보(위치) 최소화. 도시명만 있으면 날씨 조회 충분.
- 버린 대안: 현재 위치 자동(GPS) — 편하지만 위치 권한·위치정보 취급이 늘어남 → 다음 버전.
- 영향: 위치 권한 불필요. settings.city에 도시명만 저장, 날씨 API엔 도시명만 전송.

## [2026-07-05] 날씨 API = OpenWeatherMap
- 왜: 무료 티어 있고 연결이 간단해 바이브코딩 첫 연동으로 무난.
- 버린 대안: 기상청 단기예보 API — 한국 정확도는 높지만 신청·규격이 복잡.
- 영향: .env에 OPENWEATHER_API_KEY 하나. 도시명 기반 조회.
